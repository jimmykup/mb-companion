General

Watching Files and Folders

Korra will automatically enhance local HTML files for you as soon as you open them. You can adjust which kinds of files Korra enhances and which folders Korra watches for those files.
Control which local files Korra will process by adding or removing file extensions.
Separate values with commas. Default values: .html, .htm
By default Korra will attempt to process any local files that you open in your browser regardless of where they are located. If you'd prefer that Korra only watch specific folders you can add them here. Leave this field blank to allow Korra to watch all folders.
Enter the full path to the folders you want to watch. Do not include a file:// protocol.
Enter each path on its own line. You can use wildcards *.
Tip: On Mac your paths should begin with a forward slash /.
Korra doesn't watch for files on local servers out of the box (for example: http://localhost:4567/). You can add them yourself by including the appropriate path below.
Enter each path on its own line. Be sure to start each with a protocol like http://. You can use wildcards *.

Korra has several "views" available to you when you're previewing an email. Several settings can be edited to modify the default behavior on page load. And some settings can be toggled on and off while viewing an email independently of your chosen preferences here.

General

Toggle this option to overwrite the prefered color scheme of your OS. For example, if your OS is set to dark mode, but you'd prefer your emails to always be viewed as if you were in light mode. Korra will dynamically update the prefers-color-scheme CSS in your email to match your preference. You can also toggle this manually in each email using the toolbar.

Link Validation
Korra can check your links for you automatically against a list of default formatting rules and custom rules you create. Korra can also follow your links to check that they are loading properly and give you feedback on how to fix them.

General

Automatically validate the formatting of all links in an email as soon as you open it. If you disable this you can still manually check or re-check your links without reloading on a per-email basis from within the email you're previewing.
Automatically validate that all links load properly in an email as soon as you open it. If you disable this you can still manually check or re-check your links without reloading on a per-email basis from within the email you're previewing.
When enabled, if the link has a target attribute you will see an error message.
Default: On
Korra will limit certain validation checks to primary domains when they make sense. For example, validating utm parameters when the link is to a primary domain and informing you when you shouldn't use them on other links (like on third-party domains).
Enter each URL on its own line.
Prevent Korra from checking a link by adding it to the disallow list. For example, entering www.mysite.com will stop these links from being validated: www.mysite.com/index.html or www.mysite.com/path/index.html
Enter each URL on its own line.

URL Format Validation

These settings focus on validating the formatting of your links.

When enabled, if there is a link without an href attribute you will see an error message.
When enabled, if the link url is empty (href="") you will see an error message.
When enabled, if the link url ends with a '#' you will be warned.
When enabled, if the pathname portion of your link is missing a trailing '/' you will be warned. By default this will ignore links ending with common file extensions like .php, .pdf, etc.
Some email clients will assume the worst if a link goes to one place while the visible text of that link claims to go to another. Avoid being flagged as spam by turning this on to be warned so that you can make adjustments.
Default: On
Some ESP tags include characters that would normally be invalid in a link (e.g. *|tag|* or %%variable%%). Enable this to prevent Korra from potentially marking your links as invalid when it finds one.
Note: Provide the formatting that your ESP uses in order for this to work. Enter this in the "ESP Options" section.
Default: On
When enabled, Korra will warn you if your link is missing that value in the querystring.

Loading Validation

Korra can follow your links to verify that they load properly.

When enabled Korra will follow your links to verify that they load properly. If the link fails to load for some reason you'll receive an error message with the reason to help you diagnose the problem.
HEAD is quicker, but sometimes doesn't reflect the true HTTP status of a page, use GET if you're getting weird results. GET is required to be able to parse the document for an element corresponding to what is after the hashtag by ID.
Hashtags can link to elements on the destination page by their id attribute. For example, <a href="page2.html#elementID">Link</a> would link to an scroll to an element with id="elementID" when the page has loaded. Enabling this will allow Korra to automatically check that all links with hastags have a matching element. When enabled, if the page loads but the corresponding element is not found, the link will be marked with a warning.
Request Type: GET is required

If you don't want to open links in the background over and over again, enable this setting to force Korra to only load new links or links that failed to load on a previous check. Links are cached on a per-email basis. Identical links in separate emails will both be checked even if one is already cached.

Automatically clear links from cache after day(s).

You can recheck cached links at any time by clearing the cache of all links related to the email you're currently previewing from within the email preview UI. You can also clear Korra's entire cache of valid links with the button below.
